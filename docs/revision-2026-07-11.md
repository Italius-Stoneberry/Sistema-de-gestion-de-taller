# Revisión del Sistema de Gestión de Taller
**Fecha:** 11/07/2026 · **Alcance:** backend, base de datos, frontend, Docker y flujos n8n

## Resumen

El proyecto está muy bien encarado para lo que es: la base de datos no expone puertos, todas las consultas SQL usan parámetros (no encontré ninguna inyección SQL), el frontend escapa consistentemente el HTML, la descarga de adjuntos valida que la ruta quede dentro del volumen de uploads, y el asistente de WhatsApp tiene un fail-safe correcto (lista de autorizados vacía = no responde a nadie). No hay ninguna falla crítica explotable desde afuera si se respeta el modelo de despliegue (LAN + Tailscale, sin puertos abiertos a internet). Lo que sigue son los puntos a corregir, ordenados por prioridad.

---

## 1. Seguridad

### 1.1 — ALTA · Fallback inseguro del JWT_SECRET
`backend/src/auth.js` línea 3: `const SECRET = process.env.JWT_SECRET || 'dev-inseguro-cambiar'`. Si el `.env` falta o la variable queda vacía, la app arranca igual con un secreto público que está en el repo: cualquiera que lea el código puede firmarse un token de admin. Lo mismo pasa con `ADMIN_PASSWORD` en `seed.js` (fallback `admin1234`). La corrección es que el proceso se niegue a arrancar:

```js
if (!process.env.JWT_SECRET || process.env.JWT_SECRET.length < 32) {
  console.error('FALTA JWT_SECRET (mínimo 32 caracteres). Abortando.');
  process.exit(1);
}
```

### 1.2 — ALTA · El comentario de AUTORIZADOS en .env.example contradice al código
El `.env.example` dice "Vacío = permite a cualquiera (útil para probar)", pero el código hace exactamente lo contrario (asistente.js:519, lista vacía = ignora a todos, que es lo correcto). El peligro es al revés del habitual: alguien que lea ese comentario podría creer que dejarlo vacío es inofensivo, o un futuro refactor podría "arreglar" el código para que coincida con el comentario, y ahí sí cualquier cliente que escriba al WhatsApp del taller podría operar el sistema (crear trabajos, marcar cheques como cobrados, leer cuánto factura cada cliente). Corregí el comentario ya mismo: "Vacío = NO responde a nadie".

### 1.3 — MEDIA · Sin límite de intentos en el login
`/api/auth/login` no tiene rate limiting ni bloqueo por intentos (no hay `express-rate-limit` en package.json). Dentro de una LAN con invitados o un dispositivo comprometido, se puede probar contraseñas sin freno. Agregá `express-rate-limit` al login (por ejemplo 10 intentos por 15 minutos por IP) y, ya que estás, al endpoint de ingesta.

### 1.4 — MEDIA · Desactivar un usuario no lo saca del sistema
`requiereAuth` solo verifica la firma del JWT; no consulta la base. Un usuario desactivado (o al que le bajaste el rol) sigue operando con su token viejo hasta 12 horas. Para una app familiar puede ser aceptable, pero si algún día se pelea alguien o se pierde un teléfono, conviene poder cortar el acceso al instante: verificá `activo` (y el rol) contra la base en el middleware, o mantené una lista de tokens revocados en memoria.

### 1.5 — BAJA · Comparación de API key no constante en tiempo
`requiereIngest` compara con `key !== esperada`. Usá `crypto.timingSafeEqual` para eliminar el vector de timing. Es un ataque poco práctico en tu escenario, pero el fix son tres líneas.

### 1.6 — BAJA · Chequeo de ruta de adjuntos con `startsWith` sin separador
En `adjuntos.js`, `abs.startsWith(path.resolve(UPLOADS_DIR))` aceptaría `/app/uploads-otro`. Hoy no es explotable porque `archivo` lo genera el propio servidor (UUID), pero endurecelo con `startsWith(path.resolve(UPLOADS_DIR) + path.sep)`.

### 1.7 — BAJA · Contenedores corren como root
El Dockerfile no declara `USER node`, y la imagen de node corre como root por defecto. Agregá `USER node` antes del CMD (ajustando permisos de `/app/uploads`). También sumaría `no-new-privileges: true` en el compose.

### 1.8 — Consideración · Inyección de prompt vía WhatsApp
Los mensajes van crudos dentro de los prompts de Ollama. Como solo los números autorizados llegan al asistente, el riesgo real es bajo, y el diseño de borradores (`revisado = FALSE` + confirmación) ya acota el daño de una interpretación maliciosa o errónea. Mantené esa disciplina: cualquier acción nueva del asistente que escriba en la base debería nacer como borrador, como ya hacés con trabajos y cheques (los pagos hoy entran con `revisado = TRUE` directo — ver 3.4).

## 2. Bugs funcionales encontrados

### 2.1 — `express.json()` sin límite rompe las fotos por base64
`index.js:29` usa el límite default de 100 KB. Cualquier foto que n8n mande por `media_base64` a `/api/asistente/mensaje` va a superar eso y Express la rechaza con 413 antes de llegar a tu código. Si las fotos hoy entran por `media_url` no lo notaste, pero el camino base64 está muerto. Fix: `app.use(express.json({ limit: '15mb' }))`.

### 2.2 — La ingesta no acepta la disciplina "impresion"
`ingesta.js:9` tiene `DISCIPLINAS = ['laser', 'serigrafia', 'ploteo']` — quedó desactualizado respecto de la migración 006 y del resto del código. Todo trabajo que entre por n8n como "impresion" cae silenciosamente a "laser". Importá la lista desde un módulo compartido para que no vuelva a divergir.

### 2.3 — Ediciones parciales por API borran campos
En varios PUT (trabajos, cheques, contactos, pagos) los campos sin COALESCE se pisan con NULL si no vienen en el body: por ejemplo, `PUT /api/contactos/:id` con solo `{telefono}` desvincula la empresa del contacto (`empresa_id = b.empresa_id || null`). El frontend manda siempre el body completo así que hoy no se nota, pero cualquier flujo de n8n o script que haga un update parcial va a pisar datos. Definí una semántica (PATCH parcial con COALESCE en todo, o PUT que exige body completo) y aplicala pareja.

### 2.4 — Borrar un trabajo deja adjuntos huérfanos
`adjuntos.entidad_id` no tiene FK. Al eliminar un trabajo o cheque quedan las filas de adjuntos y los archivos en el volumen para siempre. Agregá la limpieza en los DELETE (borrar filas de adjuntos + `fs.unlink`), y de paso una pasada periódica que borre archivos sin fila.

## 3. Usabilidad (vista del usuario)

### 3.1 — Tablas en el celular
La familia va a usar esto sobre todo desde el teléfono por Tailscale, y las vistas principales son tablas anchas de 8-9 columnas. Es la mejora de usabilidad de mayor impacto: en pantallas angostas, renderizar cada trabajo como tarjeta (cliente + descripción arriba; badges de estado/cobro/facturación abajo; precio a la derecha) con un media query en `styles.css`. Los toques rápidos que ya tenés (avanzar estado, marcar cobro) funcionan perfecto en formato tarjeta.

### 3.2 — Confirmaciones y errores con `confirm()`/`alert()`
Los diálogos nativos del navegador cortan el flujo y en el celular se ven pobres. Un mini-sistema de toasts (15 líneas de JS) para errores y un botón "Deshacer" de 5 segundos en lugar de `confirm()` para acciones reversibles mejora mucho la sensación. Para eliminar definitivo (admin) el confirm está bien.

### 3.3 — La app como PWA
Con un `manifest.json` y un ícono, cada uno se la agrega a la pantalla de inicio del teléfono y se abre a pantalla completa como una app. Son dos archivos estáticos en `public/`, cero backend.

### 3.4 — Consistencia del asistente: los pagos no piden confirmación
Cuando anotás un cheque o trabajo por WhatsApp, el asistente pide "ok/no". Los pagos (`crearPago`) entran directo con `revisado = TRUE` y sin pregunta. Para el usuario es desconcertante que a veces pida confirmación y a veces no; unificá el flujo (todo nace borrador y se confirma con "ok").

### 3.5 — El badge de la Bandeja no se refresca solo
El contador "Bandeja (n)" se actualiza al navegar, pero si papá deja la pestaña abierta y entra algo por WhatsApp no se entera. Un `setInterval` de 60s que pegue a `/api/bandeja` y actualice el badge (y quizás el título de la pestaña) es barato y hace que la Bandeja se sienta viva.

### 3.6 — Detalles menores
El select de "Cobro" y "Facturación" en el form de trabajo podría ser un checkbox (dos toques menos). En "Trabajos de <cliente>" la tarjeta dice "Facturado total" pero suma el precio de todos los trabajos, facturados o no — o renombrala a "Total" o filtrá por `facturado`. Y las fechas se muestran como `YYYY-MM-DD`; en Argentina `DD/MM` (como ya hacés en WhatsApp con `fmtFecha`) es más natural.

## 4. Calidad de vida (mantenimiento)

### 4.1 — Backups de la base
No hay estrategia de backup y ahí vive la contabilidad del taller. Un servicio más en el compose resuelve:

```yaml
backup:
  image: prodrigestivill/postgres-backup-local:16
  restart: unless-stopped
  volumes: [ ./backups:/backups ]
  environment:
    POSTGRES_HOST: db
    POSTGRES_DB: ${DB_NAME}
    POSTGRES_USER: ${DB_USER}
    POSTGRES_PASSWORD: ${DB_PASSWORD}
    SCHEDULE: "@daily"
    BACKUP_KEEP_DAYS: 14
```

Idealmente copiá `./backups` a otro disco o a la nube de vez en cuando.

### 4.2 — Esquema duplicado entre init.sql y migraciones
`init.sql` repite el bloque de campos de IA y el de clientes que también están en `001_ia.sql` y `002_clientes.sql`. Funciona por los `IF NOT EXISTS`, pero es el típico lugar donde una instalación nueva y una migrada divergen. Opción simple: que `init.sql` tenga solo el esquema base y un script de arranque aplique las migraciones en orden con una tabla `schema_migrations`.

### 4.3 — Healthcheck y logs de la app
El contenedor `app` no tiene healthcheck (la base sí). Agregá uno contra `/api/health` para que `restart: unless-stopped` tenga señal real. Y limitá los logs de Docker (`logging: options: max-size: 10m, max-file: 3`) para que no coman disco en el NAS.

### 4.4 — Limpieza de la tabla conversaciones
`conversaciones` crece sin límite (una fila por chat, pero `datos` acumula listas). Un DELETE de filas con `actualizado_en` mayor a 30 días en el arranque alcanza.

### 4.5 — Constantes compartidas
`DISCIPLINAS`, `ESTADOS` y compañía están copiadas en trabajos.js, ingesta.js, asistente.js y app.js — así apareció el bug 2.2. Un `src/constantes.js` exportado y (si querés) un endpoint `/api/meta` para que el frontend las lea.

## 5. Optimizaciones (aprovechando la GPU local)

### 5.1 — `keep_alive` en Ollama: la mejora más grande de latencia
`ollamaJSON` no manda `keep_alive`, así que Ollama descarga el modelo a los ~5 minutos de inactividad y el próximo mensaje de WhatsApp paga la recarga completa del qwen3:14b a VRAM (varios segundos, a veces decenas). Como la GPU es dedicada a esto: `body: { ..., keep_alive: -1 }` (o `"24h"`) y el modelo queda residente. Es un cambio de una línea y transforma la respuesta del asistente.

### 5.2 — Menos llamadas al LLM por mensaje
Cada mensaje hace 2 llamadas secuenciales (clasificar → extraer detalle), y las fotos hasta 3. Dos opciones: unificar clasificación + extracción en un solo schema JSON con todos los campos nullable (una sola pasada), o usar un modelo chico (qwen3:4b o similar) solo para clasificar y dejar el 14b para la extracción. Con `keep_alive` puesto, la opción del modelo único unificado es probablemente la más simple y suficiente.

### 5.3 — Contexto de clientes en cada prompt
`contextoClientes()` mete hasta 80 nombres en cada prompt de clasificación, y hace 2 queries por mensaje. Cacheala en memoria 5 minutos (se invalida al crear empresa/contacto) y considerá mandar solo los 15-20 más recientes: menos tokens = respuesta más rápida y clasificación menos distraída.

### 5.4 — Paralelizar queries en dashboard y bandeja
`dashboard.js` hace 6 queries secuenciales y `bandeja.js` 3. Con `Promise.all` la latencia de esas pantallas baja a la de la query más lenta. Cambio mecánico, sin riesgo.

### 5.5 — Whisper
Si los audios de WhatsApp tardan, con esa GPU podés subir `ASR_MODEL` de `small` a `medium` sin problema de VRAM (conviven bien con el 14b si no corren a la vez), o quedarte en `small` que ya es correcto para español rioplatense.
